# Production Deployment Guide

Moving the NIDS from a local testing environment to a production environment requires migrating from SQLite to PostgreSQL, securing the frontend, and tuning Suricata.

## 1. Database Migration (PostgreSQL)

In production, SQLite will lock under high concurrency.
1. Update `backend/.env`:
```env
DATABASE_URL=postgresql+asyncpg://nids_user:strong_password@db_host:5432/nids_prod
```
2. Run alembic migrations against the new database:
```bash
docker-compose exec backend alembic upgrade head
```

## 2. Nginx Reverse Proxy & SSL

Do not expose the FastAPI or Next.js servers directly to the internet. Use an Nginx reverse proxy to terminate SSL and handle WebSocket upgrades.

**Example `nginx.conf`**:
```nginx
server {
    listen 443 ssl;
    server_name nids.company.local;

    ssl_certificate /etc/nginx/ssl/nids.crt;
    ssl_certificate_key /etc/nginx/ssl/nids.key;

    # Route to Next.js
    location / {
        proxy_pass http://frontend:3000;
        proxy_set_header Host $host;
    }

    # Route to FastAPI REST
    location /api/ {
        proxy_pass http://backend:8000/api/;
    }

    # Upgrade WebSockets
    location /api/v1/ws/ {
        proxy_pass http://backend:8000/api/v1/ws/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "Upgrade";
    }
}
```

## 3. High Availability

The backend is stateless (except for the WebSocket manager). You can scale the backend horizontally, but you must implement a Redis backplane for the `ConnectionManager` so that alerts generated by worker A are broadcasted to WebSockets connected to worker B.
